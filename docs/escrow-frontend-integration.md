## Contract Functions

### `depositAndLock(jobId, provider, amount, expiry)`

Called by the DC1 backend on behalf of the renter when a job is submitted.

| Param | Type | Description |
|---|---|---|
| `jobId` | `bytes32` | `keccak256` of the DC1 job UUID string |
| `provider` | `address` | Provider wallet address registered with DC1 |
| `amount` | `uint256` | USDC amount in micro-USDC (multiply by 10^6) |
| `expiry` | `uint256` | Unix timestamp ‚Äî when renter can cancel if job stalls |

**Prerequisite**: Renter must first `approve()` the escrow contract to spend their USDC.

---

### `claimLock(jobId, proof)`

Called by the provider (or DC1 backend on their behalf) after job completion.

| Param | Type | Description |
|---|---|---|
| `jobId` | `bytes32` | Same job identifier used in `depositAndLock` |
| `proof` | `bytes` | ECDSA signature from DC1 oracle over `keccak256(jobId ‚Äñ provider ‚Äñ amount)` |

**Fee split**: 75% goes to provider, 25% goes to DC1 (contract owner).

---

### `cancelExpiredLock(jobId)`

Called by the renter when job has expired without being claimed.

- Only callable by the original renter
- Only after `block.timestamp > expiry`
- Returns full USDC amount to renter

---

### `getEscrow(jobId)` (view)

Returns the `EscrowRecord` struct for any job:

```solidity
struct EscrowRecord {
    address renter;
    address provider;
    uint256 amount;
    uint256 expiry;
    EscrowStatus status; // EMPTY | LOCKED | CLAIMED | CANCELLED
}
```

## Backend Integration (ethers.js)

### Setup

```js
// backend/src/services/escrow.js
const { ethers } = require('ethers');
const EscrowABI = require('../../../contracts/abis/Escrow.json');

const provider = new ethers.JsonRpcProvider(process.env.BASE_SEPOLIA_RPC);
const oracleWallet = new ethers.Wallet(process.env.ORACLE_PRIVATE_KEY, provider);

const escrowContract = new ethers.Contract(
  process.env.ESCROW_CONTRACT_ADDRESS,
  EscrowABI.abi,
  oracleWallet
);
```

### Lock funds at job submission

```js
async function lockJobFunds(jobUuid, providerWallet, usdcAmountMicro, expiryUnix) {
  const jobId = ethers.keccak256(ethers.toUtf8Bytes(jobUuid));
  const tx = await escrowContract.depositAndLock(
    jobId,
    providerWallet,
    usdcAmountMicro,
    expiryUnix
  );
  await tx.wait();
  return tx.hash;
}
```

> **Note**: The renter must approve the escrow contract before this call succeeds.
> In the DCP flow, the backend holds USDC on behalf of renters from their pre-funded
> balance ‚Äî see the USDC approval section below.

### Sign job completion (oracle)

```js
async function signJobCompletion(jobUuid, providerWallet, usdcAmountMicro) {
  const jobId = ethers.keccak256(ethers.toUtf8Bytes(jobUuid));
  const domain = {
    name: 'DCP Escrow',
    version: '1',
    chainId: Number(process.env.NEXT_PUBLIC_BASE_CHAIN_ID || 84532),
    verifyingContract: process.env.ESCROW_CONTRACT_ADDRESS,
  };
  const types = {
    Claim: [
      { name: 'jobId', type: 'bytes32' },
      { name: 'provider', type: 'address' },
      { name: 'amount', type: 'uint256' },
    ],
  };
  const proof = await oracleWallet.signTypedData(domain, types, {
    jobId,
    provider: providerWallet,
    amount: usdcAmountMicro,
  });
  return proof;
}
```

### Release funds to provider

```js
async function releaseFundsToProvider(jobUuid, proof) {
  const jobId = ethers.keccak256(ethers.toUtf8Bytes(jobUuid));
  const tx = await escrowContract.claimLock(jobId, proof);
  await tx.wait();
  return tx.hash;
}
```

### Check escrow status

```js
async function getEscrowStatus(jobUuid) {
  const jobId = ethers.keccak256(ethers.toUtf8Bytes(jobUuid));
  const record = await escrowContract.getEscrow(jobId);
  const statusMap = ['EMPTY', 'LOCKED', 'CLAIMED', 'CANCELLED'];
  return {
    renter: record.renter,
    provider: record.provider,
    amount: record.amount.toString(),
    expiry: new Date(Number(record.expiry) * 1000).toISOString(),
    status: statusMap[record.status],
  };
}
```

## Renter Flow (end-to-end)

### Step 1: Renter tops up USDC balance

The renter deposits USDC into their DCP wallet. The DC1 backend holds this USDC
in the hot-wallet and is responsible for approving the escrow contract to spend it.

One-time approval (done by backend, not frontend):

```js
const usdcContract = new ethers.Contract(USDC_ADDRESS, ERC20_ABI, hotWallet);
await usdcContract.approve(ESCROW_CONTRACT_ADDRESS, ethers.MaxUint256);
```

### Step 2: Renter submits a job (`POST /api/jobs/submit`)

Backend:
1. Validates renter has sufficient USDC balance
2. Calculates job cost: `gpuHours √ó rate` in USDC (6 decimals)
3. Sets expiry: `now + jobTimeoutSeconds`
4. Calls `depositAndLock(jobId, providerWallet, usdcAmount, expiry)`
5. Stores escrow tx hash on the job record in SQLite

### Step 3: Job runs on provider GPU

Provider daemon polls `/api/providers/:key/jobs`, executes the Docker job, reports
completion via `POST /api/jobs/:id/complete` (or similar endpoint ‚Äî see jobs.js).

### Step 4: Backend signs and releases funds

On job completion:
1. DC1 backend calls `signJobCompletion(jobId, providerWallet, amount)` ‚Üí `proof`
2. Backend calls `claimLock(jobId, proof)` on behalf of provider
3. Contract sends 75% USDC to provider wallet, 25% to DC1 owner wallet
4. Job status updated to `completed` in SQLite

### Step 5: If job stalls (timeout/failure)

If `block.timestamp > expiry` and job was never claimed:
- Renter (or DC1 backend on their behalf) calls `cancelExpiredLock(jobId)`
- Full USDC returned to DC1 hot-wallet ‚Üí credited back to renter's off-chain balance
- Job status updated to `cancelled` in SQLite

---

## Dispute / Failure Handling

| Scenario | Resolution |
|---|---|
| Job completes, provider claims on time | `claimLock` ‚Äî 75/25 split |
| Job times out, provider never claimed | `cancelExpiredLock` ‚Äî full refund to renter |
| Provider claims but oracle proof is wrong | Contract rejects (`Invalid oracle proof`) ‚Äî funds stay locked until expiry |
| Job fails mid-run (Docker error) | DC1 backend does NOT sign; expiry passes; renter cancels |
| Dispute over job quality | Manual review by DC1 admin; backend can withhold oracle signature |

## Frontend (Next.js) Considerations

The renter dashboard does not call the contract directly. All on-chain calls are
made server-side by the Express.js backend. The frontend only needs:

1. **Display escrow status** ‚Äî call `GET /api/jobs/:id` which should include
   `escrowStatus: 'LOCKED' | 'CLAIMED' | 'CANCELLED'`

2. **Chain indicator** ‚Äî show "Secured on Base" badge using `NEXT_PUBLIC_BASE_CHAIN_ID`

3. **Transaction links** ‚Äî link to `https://sepolia.basescan.org/tx/<txHash>`
   using the escrow tx hash stored on the job record

Example component snippet:

```tsx
// In job detail component
const chainId = process.env.NEXT_PUBLIC_BASE_CHAIN_ID; // "84532"
const basescanBase = chainId === "84532"
  ? "https://sepolia.basescan.org"
  : "https://basescan.org";

{job.escrowTxHash && (
  <a href={`${basescanBase}/tx/${job.escrowTxHash}`} target="_blank">
    View on Basescan
  </a>
)}
```

## Job ID Encoding

The contract uses `bytes32` job IDs. DC1 job UUIDs (string) must be hashed before
use on-chain. Use the same encoding in both backend and any off-chain lookups:

```js
// Consistent encoding ‚Äî same on all sides
const jobId = ethers.keccak256(ethers.toUtf8Bytes(jobUuid));
```

Do NOT use `ethers.id(jobUuid)` ‚Äî it is equivalent but less explicit.

## Job Completion ‚Üí Escrow Release Flow

This section documents the exact code path triggered when a job completes, and how
the off-chain escrow hold is settled alongside the on-chain `claimLock` call.

### Backend routes that complete jobs

| Route | Caller | Escrow path |
|---|---|---|
| `POST /api/jobs/:id/result` | Provider daemon | claimLock (success) / cancelExpiredLock (failure) |
| `POST /api/jobs/:id/complete` | Admin / renter | claimLock (always ‚Äî job forced complete) |
| `POST /api/jobs/:id/fail` | Provider daemon | cancelExpiredLock (explicit failure) |

### Success path (`/result` with output, or `/complete`)

```
1. Job rows updated: status = 'completed', actual_cost_halala, provider_earned_halala
2. Provider earnings credited: claimable_earnings_halala += provider_earned
3. SQLite escrow_holds: status = 'released_provider'
4. On-chain (when ESCROW_CONTRACT_ADDRESS is set):
     chainEscrow.claimLock(jobId)
       ‚Üí Escrow.sol.claimLock(keccak256(jobId), oracleProof)
       ‚Üí 75% USDC ‚Üí provider wallet, 25% USDC ‚Üí DC1 owner wallet
5. Simulation log (when chain NOT configured):
     [escrow-sim] claimLock | jobId=<id> | providerAddress=N/A | amountHalala=<n> | dc1FeeHalala=<n>
```

### Failure path (`/result` without output, or `/fail`, or timeout)

```
1. Job rows updated: status = 'failed' / 'permanently_failed'
2. Renter balance refunded: balance_halala += cost_halala
3. SQLite escrow_holds: status = 'released_renter'
4. On-chain (when ESCROW_CONTRACT_ADDRESS is set):
     chainEscrow.cancelExpiredLock(jobId)
       ‚Üí Escrow.sol.cancelExpiredLock(keccak256(jobId))
       ‚Üí Full USDC amount returned to renter wallet
5. Simulation log (when chain NOT configured):
     [escrow-sim] cancelExpiredLock | jobId=<id> | refundHalala=<n> | reason=<reason>
```

### Enabling on-chain escrow

Set these env vars on the VPS before starting the backend via PM2:

```bash
ESCROW_CONTRACT_ADDRESS=0x...          # deployed Escrow.sol address on Base Sepolia
ESCROW_ORACLE_PRIVATE_KEY=0x...        # DC1 oracle signing key
BASE_RPC_URL=https://sepolia.base.org  # optional, this is the default
```

When both vars are set, `ChainEscrowService.isEnabled()` returns `true` and the
simulation logs are replaced with real on-chain transactions. The service is a
singleton (`getChainEscrow()`) and is imported at the top of `routes/jobs.js`.

### Provider EVM wallet (pending)

Providers do not yet have EVM wallet addresses registered in the DCP database.
Until provider wallet registration is implemented (see `providers` table schema),
the `depositAndLock` call uses the oracle wallet as the fallback `provider` address
on testnet. The simulation log reflects this with `providerAddress=N/A`.

Once the providers table gains an `evm_address` column, the escrow service should
be updated to pass `provider.evm_address` to both `depositAndLock` and as the
expected recipient in `claimLock`.

## Contract Events (for indexing / webhooks)

| Event | When emitted | Key fields |
|---|---|---|
| `Deposited(jobId, renter, provider, amount, expiry)` | On `depositAndLock` | Use to confirm lock |
| `Claimed(jobId, provider, providerAmount, feeAmount)` | On `claimLock` | Use to confirm payout |
| `Cancelled(jobId, renter, amount)` | On `cancelExpiredLock` | Use to confirm refund |
| `OracleUpdated(oldOracle, newOracle)` | On `setOracle` | Admin audit trail |

Backend can subscribe to these events using ethers.js `contract.on('Deposited', ...)`
or poll them via `contract.queryFilter(filter, fromBlock, toBlock)` as a reconciliation step.
