## Overview

This runbook provides procedures for verifying that token-based metering and billing tracking works correctly end-to-end. It covers three test scenarios:

1. **Smoke Test (Rapid Connectivity Check)** — Validates API connectivity and basic flow
2. **Full Metering Test** — Complete token tracking and database persistence validation
3. **Model Serving Validation** — Provider health and completion endpoint testing

---

## Prerequisites

### Required Credentials

- **Renter API Key** — A valid renter authentication token
  - Source: Created via `/renters/register` or from existing renter account
  - Env var: `DCP_RENTER_KEY`
  - Example: `rk-abc123def456...`

- **Admin API Token** — For accessing metering data in the database
  - Source: Generated by admin (server operator)
  - Env var: `DC1_ADMIN_TOKEN`
  - Example: `admin-xyz789...`

- **Provider URL** (optional, for provider-side testing)
  - Format: `http://<provider-host>:<port>`
  - Example: `http://10.0.0.50:8000` or `https://gpu-1.provider.local`

### Renter Balance Requirement

- Minimum balance: **100 halala** (0.001 USDC equivalent)
- Used for: Test job submission and token charges
- Check current balance: `curl -s "$DCP_API_BASE/renters/me?key=$DCP_RENTER_KEY" | jq .renter.balance_halala`

### API Accessibility

- Backend API must be reachable at `DCP_API_BASE` (default: `http://76.13.179.86:8083/api`)
- Health check: `curl -s $DCP_API_BASE/health | jq .`
- Expected: `{"status":"ok","timestamp":"...","version":"..."}`

## Test 1: Smoke Test (5 minutes)

### Purpose

Rapid validation that the API is reachable and metering flow works end-to-end.

### Steps

1. **Set environment variables:**
   ```bash
   export DCP_API_BASE=https://api.dcp.sa
   export DCP_RENTER_KEY="<your-renter-api-key>"
   export DC1_ADMIN_TOKEN="<your-admin-token>"
   ```

2. **Run the smoke test:**
   ```bash
   node scripts/vllm-metering-smoke.mjs
   ```

3. **Expected output:**
   ```
   === 1) API Connectivity Check ===
   [PASS] API health check - HTTP 200

   === 2) Renter Preconditions ===
   [PASS] Renter key valid - renter_id=renter-abc123, balance_halala=5000
   [PASS] Sufficient renter balance - balance=5000

   === 3) vLLM /complete Request ===
   [PASS] vLLM completion succeeded - usage: 23p + 7c = 30t
   [PASS] Token counts present - total_tokens=30

   === 4) Verify serve_sessions Metering (Admin API) ===
   [PASS] Serve session found in database - session_id=sess-xyz789
   [PASS] Token counts persisted - total_tokens=30 (expected > 0)
   [PASS] Cost calculated and tracked - total_billed_halala=150 halala (expected > 0)
   [PASS] Last inference timestamp updated - last_inference_at=2026-03-24T11:20:45.123Z

   === 5) Billing Accuracy Checkpoint ===
   [PASS] Serve sessions created on submit - serve_sessions record created with job_id at vLLM job submission
   [PASS] Token rate lookup and application - cost_halala = total_tokens × token_rate_halala (from cost_rates table)

   Summary
   Checks passed: 9/9
   Checks failed: 0
   ```

4. **Check exit code:**
   ```bash
   echo "Exit code: $?"
   ```
   - `0` = All checks passed ✓
   - `1` = Authentication failed ✗
   - `2` = vLLM dispatch failed ✗
   - `3` = Metering/database verification failed ✗

### Pass/Fail Criteria

| Check | Pass Criteria | Failure Indicates |
|-------|--------------|-------------------|
| API Health | HTTP 200 | Backend unreachable or down |
| Renter Valid | Renter ID and balance present | Invalid renter key |
| Sufficient Balance | Balance >= 100 halala | Renter account underfunded |
| vLLM Completion | HTTP 200, valid token counts | Model serving failed or timeout |
| Serve Session Found | Session ID in response | Database persistence issue |
| Tokens Persisted | total_tokens > 0 | Token update failed in DB |
| Cost Calculated | total_billed_halala > 0 | Cost calculation failed |
| Timestamp Updated | last_inference_at is not null | Timestamp logic broken |

## Test 3: Model Serving Validation (per provider, 2-3 minutes)

### Purpose

Validate that a provider endpoint is healthy and can serve model completions.

### Prerequisites

- Provider URL (e.g., `http://10.0.0.50:8000`)
- Model ID running on that provider (e.g., `TinyLlama/TinyLlama-1.1B-Chat-v1.0`)

### Steps

1. **Run the validation script:**
   ```bash
   node scripts/validate-model-serving.mjs \
     --provider-url http://10.0.0.50:8000 \
     --model TinyLlama/TinyLlama-1.1B-Chat-v1.0 \
     --timeout 30000
   ```

2. **Expected output:**
   ```
   Model Serving Validation Suite
   Provider URL: http://10.0.0.50:8000
   Model ID: TinyLlama/TinyLlama-1.1B-Chat-v1.0
   Timeout: 30000ms

   === 1) Health Check ===
   [PASS] GET /health - HTTP 200, 45ms

   === 2) Model Completions Test ===
   [PASS] POST /v1/completions - HTTP 200, 2150ms, 5 tokens
   [PASS] Response structure valid - 1 choice(s) returned
   [PASS] Model name in response - model=TinyLlama/TinyLlama-1.1B-Chat-v1.0

   === 3) Performance Analysis ===
   [PASS] Completion latency acceptable - 2150ms (threshold: 10000ms)

   Summary
   Checks passed: 5/5
   Checks failed: 0

   Provider Validation Details:
   - Health check latency: 45ms
   - Completion latency: 2150ms
   - Model ID: TinyLlama/TinyLlama-1.1B-Chat-v1.0
   - Response tokens: 5
   ```

3. **Check exit code:**
   ```bash
   echo "Exit code: $?"
   ```

### Exit Codes

| Code | Meaning |
|------|---------|
| 0 | All checks passed |
| 1 | Invalid arguments |
| 2 | Health check failed |
| 3 | Completions endpoint error |
| 4 | Latency threshold exceeded |

### Pass/Fail Criteria

| Check | Pass | Fail |
|-------|------|------|
| Health endpoint | HTTP 200, <1000ms | HTTP 5xx, timeout, unreachable |
| Completions endpoint | HTTP 200, valid response | HTTP 4xx/5xx, invalid JSON, timeout |
| Response structure | `choices` array with ≥1 item | Missing/empty choices |
| Latency | <10 seconds | ≥10 seconds |

## Database Inspection

### Check Token Persistence Directly

After running the smoke test, verify tokens were written to the database:

```bash
# Connect to DCP database (requires admin access)
psql postgresql://admin:password@db.internal/dcp << 'EOF'

-- List all serve_sessions from the last hour
SELECT
  id, job_id, renter_id, model_id,
  total_tokens, total_inferences, total_billed_halala,
  created_at, last_inference_at
FROM serve_sessions
WHERE created_at > now() - interval '1 hour'
ORDER BY created_at DESC
LIMIT 10;

EOF
```

### Expected Results

- **total_tokens** > 0 (actual count from vLLM response)
- **total_inferences** = 1 (for a single request)
- **total_billed_halala** > 0 (tokens × token_rate_halala from cost_rates)
- **last_inference_at** = timestamp of request completion

## Common Issues & Troubleshooting

### Issue: "Cannot load renter profile" (HTTP 401/403)

**Cause:** Invalid or expired renter API key
**Fix:**
1. Verify `DCP_RENTER_KEY` is set correctly
2. Check renter exists: `curl -s $DCP_API_BASE/renters/me?key=$DCP_RENTER_KEY | jq .`
3. If not found, create new renter via admin panel

---

### Issue: "Renter balance too low"

**Cause:** Insufficient halala balance
**Fix:**
1. Check balance: `curl -s "$DCP_API_BASE/renters/me?key=$DCP_RENTER_KEY" | jq .renter.balance_halala`
2. If < 100, fund the account via admin panel or test faucet
3. Retry test

---

### Issue: "vLLM complete failed: HTTP 500"

**Cause:** Model serving or backend error
**Fix:**
1. Check if provider is online: `node scripts/validate-model-serving.mjs --provider-url http://provider:8000 --model <model-id>`
2. Check backend logs: `ssh admin@76.13.179.86 "pm2 logs dc1-provider-onboarding | tail -50"`
3. Verify model is available in cost_rates table

---

### Issue: "Serve session found in database: HTTP 404"

**Cause:** Admin API not available or token not persisted
**Fix:**
1. Verify `DC1_ADMIN_TOKEN` is set and valid
2. Check if admin endpoints are enabled: `curl -H "Authorization: Bearer $DC1_ADMIN_TOKEN" $DCP_API_BASE/admin/health`
3. Verify serve_sessions table exists and has entries: `psql ... -c "SELECT COUNT(*) FROM serve_sessions;"`

---

### Issue: "Timeout: Request exceeded 30000ms"

**Cause:** Provider or backend is slow or not responding
**Fix:**
1. Increase timeout: `--timeout 60000` (60 seconds)
2. Check provider CPU/memory: `ssh provider "top -b -n 1 | head -20"`
3. Check network latency: `ping -c 3 provider-ip`
4. Restart provider service if needed

## Continuous Monitoring

### Automated Health Check (Cron)

Add to crontab to run smoke test every 15 minutes:

```bash
*/15 * * * * \
  DCP_API_BASE=https://api.dcp.sa \
  DCP_RENTER_KEY="..." \
  DC1_ADMIN_TOKEN="..." \
  node /home/node/dc1-platform/scripts/vllm-metering-smoke.mjs \
  >> /var/log/dcp-metering-smoke.log 2>&1
```

Parse results:

```bash
tail -f /var/log/dcp-metering-smoke.log | grep -E '(PASS|FAIL|Exit code)'
```
