## Renter onboarding checklist

1. Register at `/renter/register` and copy your `dcp-renter-...` key.
2. Top up wallet from `/renter/billing`.
3. Pick a live provider from `/renter/marketplace`.
4. Submit a workload from `/renter/playground` or API.
5. Track status and retrieve output.

## API-first quick flow

### 1) Validate renter key

```bash
curl -s "https://dcp.sa/api/dc1/renters/me?key=<RENTER_KEY>"
```

### 2) Browse available GPUs

```bash
curl -s "https://dcp.sa/api/dc1/renters/available-providers"
```

### 3) Submit workload

```bash
curl -s -X POST "https://dcp.sa/api/dc1/jobs/submit" \
  -H "Content-Type: application/json" \
  -H "x-renter-key: <RENTER_KEY>" \
  -d '{
    "provider_id": 42,
    "job_type": "llm_inference",
    "duration_minutes": 3,
    "container_spec": { "image_type": "vllm-serve" },
    "params": {
      "model": "TinyLlama/TinyLlama-1.1B-Chat-v1.0",
      "prompt": "Explain Saudi GPU marketplaces in 3 bullets"
    }
  }'
```

### 4) Poll output

```bash
curl -s "https://dcp.sa/api/dc1/jobs/<JOB_ID>/output?key=<RENTER_KEY>"
```

Optional status check:

```bash
curl -s -H "x-renter-key: <RENTER_KEY>" "https://dcp.sa/api/dc1/jobs/<JOB_ID>"
```

## Billing behavior

- DCP uses halala internally (`100 halala = 1 SAR`).
- Estimated cost is reserved at submit time.
- Final settlement is calculated against actual runtime.
- Unused reserved balance is released after final settlement.

## Status model

Typical progression:

`pending -> queued -> running -> completed`  
(or terminal `failed` / `cancelled`)
